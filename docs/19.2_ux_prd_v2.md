## §0 — THÊM MỚI: Hybrid Semantic Search trong Phase 0

**Vấn đề PRD-16:** Phase 0 chỉ dùng token-match. Input như "da bị ăn vào", "bị nặng đầu không làm được gì", "hay quên lung tung" sẽ trả về score gần 0 vì không khớp token nào.

**Sửa: Embedding search là primary, token-match là fallback**

```
Phase 0 — Luồng mới:

User nhập: "da bị ăn vào"
      ↓
[Song song]
  ┌── A: Embedding Search ─────────────────────────────── ┐
  │  Browser embed query (Transformers.js, ~55ms)          │
  │  Hamming search trong k02_symptom.bin (~5ms)           │
  │  → Top 15 symptoms theo semantic similarity            │
  └────────────────────────────────────────────────────────┘
  ┌── B: Token-Match (luôn chạy, PHP) ─────────────────── ┐
  │  Tokenize + search name_vi + aliases + notes           │
  │  → Top 15 symptoms theo token score                    │
  └────────────────────────────────────────────────────────┘
      ↓
[Merge Results]
  - Nếu embedding sẵn sàng: weight embedding 0.7 + token 0.3
  - Nếu model chưa load:    dùng token-match 100%
  - Dedup theo symptom_code, giữ score cao nhất
  → Top 12 đi vào Symptom Picker

[Song song] K01 search:
  - Embedding search trong k01_phrase.bin
  - Nếu similarity > 0.75 → coi là K01 match → Nhánh A
  - Nếu không → Nhánh B (Confirmation Step)
```

**Kết quả so sánh:**

| Input | Token-match | Embedding search |
|-------|------------|-----------------|
| "da bị ăn vào" | Không match | pressure_ulcer_risk (0.81) |
| "bị nặng đầu" | Kém | heavy_dull_headache (0.84) |
| "hay quên" | Thiếu | cognitive_decline (0.87) |
| "đau đầu" | Tốt | throbbing_headache (0.89) — cả hai tốt |
| "loét thịt" | Trung bình | pressure_ulcer_risk (0.85) |

**Performance targets:**
- Model load (lần đầu): ~3s (cached vào browser)
- Embed query: ~55ms
- Hamming search (345 docs): <2ms
- Tổng Phase 0: <100ms (sau khi model đã load)

## §1 — SỬA: Nhánh B (K01 No-Match Flow)

**Vấn đề PRD-16:** Khi K01 không match, PRD-16 ghi "dùng semantic top-5 làm anchor, bỏ qua disambiguate, vào thẳng Picker." Đây là cách xử lý sai vì top-5 semantic có thể là 5 triệu chứng từ 5 hệ cơ quan khác nhau, không coherent. Ngoài ra, OrganScore trong SymptomRanker (tín hiệu W2) mất signal vì không có yhct_hint.

**Sửa thành: Nhánh B có Confirmation Step riêng**

```
Nhánh B — Luồng mới:

BƯỚC B1 — Hiển thị top 5 kết quả semantic dưới dạng CÂU HỎI XÁC NHẬN:
  ┌─────────────────────────────────────────────────────┐
  │ Bạn vừa mô tả: "tôi già, bị loét thịt do nằm lâu"  │
  │                                                     │
  │ Dựa trên mô tả, hệ thống hiểu bạn đang gặp vấn đề  │
  │ về (chọn TẤT CẢ đúng với bạn):                      │
  │                                                     │
  │ ☑ Vết loét da do nằm lâu (loét tỳ đè)              │
  │ ☐ Đau nhức xương khớp do tuổi già                   │
  │ ☐ Mệt mỏi, yếu sức toàn thân                       │
  │ ☐ Mất kiểm soát đại tiểu tiện                       │
  │ ☐ Lưng mỏi, gối yếu                                 │
  │                                                     │
  │ ☐ Không có cái nào — Tôi muốn mô tả lại             │
  │                                                     │
  │  [← Mô tả lại]    [Tiếp theo →]                     │
  └─────────────────────────────────────────────────────┘

BƯỚC B2 — Nếu user chọn ≥ 1:
  → Các mục được chọn = anchor_codes
  → Với mỗi anchor, chạy mini-disambiguation:
    "Vết loét da của bạn trông như thế nào?"
    [A] Vùng da đỏ, chưa bị loét (giai đoạn 1)
    [B] Da bị trợt, phồng rộp (giai đoạn 2)
    [C] Vết loét sâu, nhìn thấy mô mỡ/cơ (giai đoạn 3-4)
    [D] Vết loét có mùi, chảy dịch, có thể nhiễm trùng
  → Đây trở thành anchor_codes + yhct_hint (gốc = user's own description)
  → Vào Picker với anchor đã xác nhận

BƯỚC B2 — Nếu user chọn "Không có cái nào":
  → Clear input, hiển thị lại ô nhập với gợi ý:
    "Hãy mô tả triệu chứng cụ thể hơn, ví dụ:
     'đau đầu kèm buồn nôn', 'mệt mỏi sau khi ăn', 'khó thở khi leo cầu thang'"
```

**Tại sao thiết kế này tốt hơn:**
- Xác nhận với user thay vì giả định
- Cung cấp yhct_hint từ confirmation → SymptomRanker có đủ signal W2
- Mini-disambiguation ngắn (1 câu) nhưng đủ để phân loại severity
- Bảo vệ khi semantic search fail (user có thể mô tả lại)

## §2 — THÊM MỚI: Pertinent Negatives (Triệu Chứng Âm Tính Mang Tính Sống Còn)

**Vấn đề:** UI Symptom Picker chỉ cho phép tick những gì user CÓ. Những gì user KHÔNG CÓ quan trọng không kém để loại trừ bệnh nguy hiểm.

Ví dụ: Đau đầu mà KHÔNG cứng gáy, KHÔNG sốt → loại trừ viêm màng não.
Nếu user không tick "cứng gáy", hệ thống không biết user không bị hay chỉ không đọc đến dòng đó.

**Thiết kế: Rule-out Screen (xuất hiện sau Picker, trước Quick Questions)**

```
Hiển thị ĐIỀU KIỆN: chỉ khi top hypothesis có associated red flags

┌─────────────────────────────────────────────────────┐
│ Bước 3.5/5  |  Xác nhận dấu hiệu nguy hiểm          │
│                                                     │
│ 🛡 Để đảm bảo an toàn, xin xác nhận bạn KHÔNG CÓ   │
│ các dấu hiệu sau:                                   │
│                                                     │
│ ┌── Dựa trên: "Đau đầu dữ dội" ─────────────────── ┐│
│ │                                                   ││
│ │ Tôi KHÔNG có:                                     ││
│ │ ☐ Cứng gáy (không cúi đầu được bình thường)      ││
│ │ ☐ Sốt cao (trên 38.5°C)                          ││
│ │ ☐ Sợ ánh sáng dữ dội, phải nhắm mắt              ││
│ │                                                   ││
│ │ ⚠ Nếu có BẤT KỲ dấu hiệu nào trên đây,          ││
│ │   hãy TICK vào Symptom Picker thay vì xác nhận   ││
│ │   ở đây.                                         ││
│ └───────────────────────────────────────────────── ┘│
│                                                     │
│ [← Quay lại kiểm tra]    [Xác nhận không có →]     │
└─────────────────────────────────────────────────────┘
```

**Logic xác định khi nào hiển thị Rule-out Screen:**

```php
// Lấy red flag satellite symptoms của top hypothesis
$topPattern = $hypothesisEngine->getTopHypothesis($sessionData);
$rfSatellites = RedFlagSatelliteService::getSatellites($topPattern, $k05Rules);

if (!empty($rfSatellites) && count($rfSatellites) >= 2) {
    // Chèn bước 3.5 vào flow
    $session->set('show_ruleout_screen', true);
    $session->set('ruleout_symptoms', $rfSatellites);

```

**Kết quả: nếu user tick là CÓ một triệu chứng "âm tính":**
```
→ Thêm symptom đó vào selected_codes
→ Re-rank lại Picker
→ Có thể trigger red flag escalation
```

## §3 — SỬA: Red Flag Satellite Promotion trong Re-ranking

**Vấn đề PRD-16:** Khi user tick "đau đầu dữ dội đột ngột", adaptive re-ranking đẩy co-occurring symptoms lên. Nhưng không ưu tiên đặc biệt các satellite symptoms của red flag (cứng gáy, sốt, sợ ánh sáng) để confirm/exclude SAH.

**Sửa: Thêm Signal W5 vào RelevanceScore**

```
RelevanceScore(symptom) =
    W1 × AnchorScore
  + W2 × OrganScore
  + W3 × CoOccurrenceScore
  + W4 × CategoryScore
  + W5 × RedFlagSatelliteScore    ← MỚI, ghi đè các signal khác

W1=0.35, W2=0.20, W3=0.20, W4=0.10, W5=0.15
```

**RedFlagSatelliteScore:**
```php
private function redFlagSatelliteScore(string $symptomCode, array $selectedCodes): float {
    foreach ($selectedCodes as $selected) {
        // Kiểm tra xem selected symptom có phải RF trigger không
        $rfRule = $this->k05Rules->findBySatelliteSymptom($selected);
        if (!$rfRule) continue;

        // Kiểm tra xem symptom hiện tại có phải satellite của RF đó không
        if (in_array($symptomCode, $rfRule['satellite_symptoms'] ?? [])) {
            return 1.0;  // Max score — satellite của red flag
        }
    }
    return 0.0;
}
```

**Display đặc biệt trong Picker:**
```
Khi W5 > 0 cho một symptom:
  → Hiển thị ở TOP của Nhóm 1, trước cả anchor symptoms
  → Label đặc biệt: "⚠ Quan trọng cần xác nhận"
  → Màu nền vàng nhạt
  → Không thể bị cuộn qua (sticky top nếu mobile)
```

## §4 — SỬA: Quick Questions — Nâng từ 3 lên 5 Câu

**Vấn đề PRD-16:** 3 câu hỏi hiện tại (lưỡi, cảm xúc, functional impact) thiếu:
1. Onset/duration → cần cho K06 time-gated clusters
2. Trajectory → cần để detect "đang xấu nhanh"
3. Thuốc đang dùng → cần cho K07 herb-drug check
4. Prior treatment → cần cho YHCT reasoning và drug interaction risk

**Thiết kế mới: 5 câu hỏi**

**Mapping kết quả:**

| Câu | Giá trị | Tác động |
|-----|---------|---------|
| Onset = "4-7 ngày" + fever | → Trigger Dengue cluster check (K06 time-gated) |
| Trajectory = "Xấu rất nhanh" | → Flag L2 review, hiển thị "Nếu xấu hơn trong 2 giờ → đến ER" |
| Lưỡi = Đỏ | → +delta trong YHCTEngine (nhiệt chứng) |

- Câu hỏi lưỡi vẫn có icon màu minh họa

## §5 — THÊM MỚI: Contradiction Detection và Over-selection Warning

**Vấn đề:** Không có cơ chế khi user tick mâu thuẫn ("sợ lạnh" + "sợ nóng") hoặc tick quá nhiều (25-30 triệu chứng).

**Thiết kế:**

```javascript
// Client-side, chạy sau mỗi lần tick

const CONTRADICTIONS = [
    ['cold_aversion', 'heat_aversion'],         // sợ lạnh vs sợ nóng
    ['excessive_sweating', 'no_sweating'],       // ra mồ hôi nhiều vs không ra mồ hôi
    ['excessive_appetite', 'poor_appetite'],     // ăn nhiều vs ăn ít
    ['constipation_hard_stools', 'loose_watery_stools'],  // táo bón vs tiêu lỏng
    ['cold_extremities', 'hot_palms_soles'],     // tay chân lạnh vs lòng bàn tay nóng
];

// Detect contradictions
function detectContradictions(selectedCodes) {
    for (const [a, b] of CONTRADICTIONS) {
        if (selectedCodes.includes(a) && selectedCodes.includes(b)) {
            return { conflict: true, pair: [a, b] };
        }
    }
    return { conflict: false };
}

// Over-selection
function checkOverSelection(selectedCodes) {
    return selectedCodes.length > 15;
}
```

## §5.5 — THÊM MỚI: "Câu Hỏi Cần Hỏi" với Triệu Chứng X

**Yêu cầu mới:** Khi user chọn (hoặc hover) một triệu chứng trong Picker, hệ thống hiển thị các câu hỏi follow-up liên quan — giúp user xác nhận triệu chứng đó đúng hơn và cung cấp thêm context cho engine.

**Nguồn dữ liệu:** K04 `key_questions_for_confirmation` đã được embed vào `k04_question.bin`.

**Thiết kế UX:**

```
User chọn "Đau đầu nhịp đập một bên":
┌─────────────────────────────────────────────────────┐
│ ℹ Để mô tả chính xác hơn:                          │
│                                                     │
│ ☑ Đau thường xảy ra một bên (trái hoặc phải)?      │
│   [Phải] [Trái] [Hai bên xen kẽ] [Không rõ]        │
│                                                     │
│ ☑ Đau kéo dài bao lâu mỗi lần?                     │
│   [< 30 phút] [Vài giờ] [Cả ngày] [Vài ngày]       │
│                                                     │
│ ☑ Đau tệ hơn khi:                                  │
│   [☐ Di chuyển] [☐ Ánh sáng] [☐ Tiếng ồn]          │
│                                                     │
│ [Lưu mô tả này]              [Bỏ qua →]            │
└─────────────────────────────────────────────────────┘
```

**Logic:**

```javascript
// Khi user click vào symptom card:
async function onSymptomSelected(symptomCode, symptomNameVi) {
    // Search câu hỏi liên quan qua embedding
    const questions = await window.semanticSearch.search(
        symptomNameVi, 'k04_question', 5
    );

    // Chỉ hiển thị nếu có câu hỏi với similarity > 0.60
    const relevant = questions.filter(q => q.similarity > 0.60);
    if (relevant.length >= 2) {
        showQuestionTooltip(symptomCode, relevant);
    }
}

// Câu trả lời được lưu vào session như sub-symptoms:
// "đau đầu_one_side" + "đau đầu_worse_light" → thêm vào selectedCodes
// → Re-rank Picker với codes mới
```

**MVP simplification:**
- Tooltip chỉ hiển thị text câu hỏi (không cần input options)
- User tự đánh dấu Yes/No một lần → tăng/giảm symptom score
- Không bắt buộc — user có thể close tooltip và tiếp tục

## §6 — SỬA: Nhóm 3 dùng K03 Differential Symptoms

**Vấn đề PRD-16:** Nhóm 3 ("Để chẩn đoán chính xác hơn") lấy supporting symptoms của top candidate. Đây là cách thụ động. Cách đúng: khi có 2 candidates đang cạnh tranh, dùng K03.differentiates_from để hỏi chính xác triệu chứng phân biệt chúng.

**Sửa logic Nhóm 3:**

```php
function buildGroup3(array $competingHypotheses, array $k03Rules, array $k04Patterns): array {
    // Trường hợp 1: 2+ candidates đang cạnh tranh (score cách nhau < 20%)
    if (count($competingHypotheses) >= 2) {
        $top1 = $competingHypotheses[0];
        $top2 = $competingHypotheses[1];

        // Tìm symptoms phân biệt top1 vs top2
        $differentiators = HypothesisEngine::getDifferentiatingSymptoms(
            [$top1, $top2], $k03Rules, $k04Patterns
        );

        return [
            'title'    => "Để xác định chính xác giữa \"{$top1['name_vi']}\" và \"{$top2['name_vi']}\":",
            'label'    => '🔬 Câu hỏi phân biệt',
            'symptoms' => $differentiators,
            'type'     => 'differential',
        ];
    }

    // Trường hợp 2: Chỉ có 1 candidate rõ ràng
    // → Dùng key_questions_for_confirmation từ K04
    $topPattern = $competingHypotheses[0];
    $confirmQuestions = $k04Patterns[$topPattern['pattern_id']]['key_questions_for_confirmation'] ?? [];

    return [
        'title'    => "Để xác nhận \"{$topPattern['name_vi']}\":",
        'label'    => '✓ Triệu chứng xác nhận',
        'symptoms' => array_column($confirmQuestions, 'symptom_code'),
        'type'     => 'confirmation',
    ];
}
```

**Kết quả hiển thị:**

```
┌── 🔬 Câu hỏi phân biệt ────────────────────────────┐
```

## §8 — THÊM MỚI: CC-Type Routing

**Vấn đề:** Tất cả input đều vào cùng một Symptom Picker. Một số loại input cần flow khác.

**Thiết kế CC-Type detection (Phase 0 extension):**

```php
// Chạy sau tokenize, trước K01 match check

const CC_TYPES = [
    'CC_GOAL'       => ['giảm cân', 'tăng cân', 'muốn khỏe', 'tăng sức đề kháng', 'bổ dưỡng'],
    'CC_FOLLOWUP'   => ['đang điều trị', 'tái khám', 'đã uống thuốc', 'sau phẫu thuật'],
    'CC_ANXIETY'    => ['sợ bị ung thư', 'sợ bị AIDS', 'lo bị bệnh', 'kiểm tra xem có bệnh không'],
    'CC_PROXY'      => ['con tôi', 'mẹ tôi', 'bố tôi', 'chồng tôi', 'vợ tôi', 'ông nội', 'bà ngoại'],
    'CC_ACUTE'      => null,  // default — không match gì ở trên
];

function detectCCType(array $tokens, string $rawInput): string {
    foreach (CC_TYPES as $type => $keywords) {
        if ($keywords === null) continue;
        foreach ($keywords as $kw) {
            if (mb_strpos($rawInput, $kw) !== false) return $type;
        }
    }
    return 'CC_ACUTE';  // default
}
```

**Flow thay đổi theo CC-Type:**

| CC-Type | Flow thay đổi |
|---------|--------------|
| CC_GOAL | Bỏ qua red flag check standard; route vào constitutional assessment |
| CC_FOLLOWUP | Hỏi thêm: "Bạn đang điều trị bệnh gì?" → dùng làm context |
| CC_ANXIETY | Hiển thị reassurance banner trước Picker; kết quả page có anxiety-aware language |
| CC_PROXY | Hiển thị "Bạn đang hỏi cho ai?" → điều chỉnh gender/age assumptions |
| CC_ACUTE | Flow chuẩn (PRD-16) |

**MVP: Chỉ implement CC_PROXY và CC_ACUTE cho Sprint 1.**
CC_GOAL, CC_FOLLOWUP, CC_ANXIETY → Phase 2.
